## 第一章：ReAct 范式与 Agent 推理引擎

### 面试题 1：请简述 ReAct 范式的核心思想，并说明在本项目中是如何实现的？

**面试话术：**

"面试官您好，ReAct 范式的核心思想是让智能体在解决问题时，显式地进行'思考-执行'循环。具体来说，就是先分析当前状态，决定下一步行动（这是思考阶段），然后调用工具获取外部信息（这是执行阶段），获取结果后再进行总结。

在我们项目中，我们使用了 LangChain 的 `create_tool_calling_agent` 来创建工具调用型 Agent，并通过 `AgentExecutor` 来管理整个推理流程。关键在于配置 `return_intermediate_steps=True`，这样我们就能获取每一步的思考内容和工具调用结果。

具体实现上，我们在 `agent.py` 中创建了 `get_agent_stream_response` 函数，这个函数会异步流式执行 Agent。当获取到 `intermediate_steps` 时，我们会记录每一步的 thought、调用的工具、工具输入和工具输出。这样就能实现一个完整的 Think-Execute 循环。

举个实际的例子，当用户提问时，Agent 会先分析问题，决定是否需要调用 RAG 检索工具，如果需要，就调用 `rag_summary_tools`，获取检索结果后再进行总结回答。整个过程是异步的，支持流式输出，用户可以实时看到思考过程和最终答案。"

**代码参考：**

```python
# 文件路径: backend/app/agent/agent.py
async def get_agent_stream_response(query, session_id, user_id):
    agent_executor = agent_factory.create_agent_executor()
    
    async for chunk in agent_executor.astream({
        "input": query,
        "chat_history": chat_history,
        "system_prompt": system_prompt
    }):
        if "intermediate_steps" in chunk:
            for action, observation in chunk["intermediate_steps"]:
                steps.append({
                    "thought": action.log,
                    "tool": action.tool,
                    "tool_input": action.tool_input,
                    "tool_output": observation
                })
        elif "output" in chunk:
            yield chunk["output"]
```

### 面试题 2：项目中 Agent 的工具定义机制是怎样的？如何实现工具调用链的管理？

**面试话术：**

"在我们项目中，工具定义主要通过 LangChain 的 `@tool` 装饰器来实现。每个工具就是一个异步函数，通过装饰器可以自动提取函数的 `description` 作为工具说明，供 Agent 在决策时使用。

比如我们定义了 `rag_summary_tools` 工具，它接收用户查询和用户 ID，调用 RAG 服务获取文档摘要。这里有个细节是，我们使用 `ContextVar` 来传递当前用户 ID，这样即使工具没有显式传入用户 ID，也能从上下文中获取。

工具调用链的管理主要通过 `AgentFactory` 工厂模式来实现。工厂类会维护一个默认工具列表，包括检索工具、时间工具、天气工具等。当创建 `AgentExecutor` 时，会将这些工具注入到 Agent 中。Agent 在执行时会根据问题自动选择合适的工具进行调用。

调用链的流程是这样的：用户提问 → Agent 分析 → 决定调用哪个工具 → 执行工具获取结果 → 根据结果决定是否继续调用其他工具或直接总结回答。整个过程是自动的，Agent 会根据工具执行结果进行多轮推理。"

**代码参考：**

```python
# 文件路径: backend/app/agent/agent_tools.py
from langchain_core.tools import tool

@tool(description="用于从向量数据库里检索文档并生成摘要")
async def rag_summary_tools(query: str, user_id: str = None) -> str:
    effective_user_id = user_id or get_current_user_id_from_context()
    result = await RagService(effective_user_id).get_documents_and_summary(query)
    return f"摘要: {result['summary']}\n文档: {result['documents']}"

# 文件路径: backend/app/agent/agent.py
class AgentFactory:
    def _get_default_tools(self) -> List[BaseTool]:
        return [rag_summary_tools, get_weather_tools, what_time_is_now]
    
    def create_agent_executor(self, custom_tools=None):
        tools = custom_tools or self.default_tools
        agent = create_tool_calling_agent(chat_model, tools, prompt)
        return AgentExecutor(agent=agent, tools=tools)
```

### 面试题 3：Agent 的多轮推理状态是如何管理的？会话历史如何维护？

**面试话术：**

"面试官您好，我们项目中的多轮推理状态管理主要通过三个层面来实现：

首先是用户上下文管理，我们使用 Python 的 `ContextVar` 来存储当前用户 ID。这样在整个请求生命周期中，任何地方都可以获取到当前用户，而不需要在每个函数中显式传递。

其次是会话历史的维护，我们将历史消息存储在 MySQL 数据库中。当用户发起请求时，会从数据库加载该会话的历史消息，然后构建成 LangChain 的 `ChatMessage` 对象列表，包括 `HumanMessage` 和 `AIMessage`，传递给 Agent 作为上下文。

第三个层面是中间步骤的记录，每次 Agent 执行时，我们都会记录每一步的思考内容和工具调用信息，这些信息可以用于调试和展示 Agent 的推理过程。

具体来说，在 `get_agent_stream_response` 函数中，我们先通过 `session_manager.get_history` 获取历史消息，然后转换为 `ChatMessage` 列表，最后传递给 `agent_executor.astream` 方法进行流式执行。执行过程中，我们会实时记录 `intermediate_steps`，这样就能完整地追踪整个推理过程。"

**代码参考：**

```python
# 文件路径: backend/app/agent/agent_tools.py
current_user_id_var: ContextVar[str] = ContextVar('current_user_id', default=None)

def set_current_user_id(user_id: str):
    current_user_id_var.set(user_id)

# 文件路径: backend/app/agent/agent.py
async def get_agent_stream_response(query, session_id, user_id):

    chat_history: List[BaseMessage] = []
    for user_msg, assistant_msg in history:
        chat_history.append(HumanMessage(content=user_msg))

```

## 第二章：RAG 知识库检索增强生成

### 面试题 4：请详细说明项目中 Hybrid Retriever 的实现原理，包括 BM25 和向量检索的融合策略？

**面试话术：**

"在我们的项目中，Hybrid Retriever 是实现高效检索的核心组件。它结合了两种检索策略：BM25 和向量检索。

BM25 是基于词频-逆文档频率的关键词检索方法，擅长精确匹配，对于短查询效果很好。而向量检索是基于语义相似度的检索，能够理解上下文语义，对于长查询和语义理解更擅长。

我们的融合策略是动态调整权重。具体来说，我们会根据查询长度来动态分配权重：对于短查询（少于20字），我们给 BM25 更高的权重（0.7），向量检索较低（0.3）；对于长查询（超过50字），我们给向量检索更高的权重（0.7），BM25 较低（0.3）；中等长度的查询则各占 50%。

实现上，我们使用 LangChain 的 `EnsembleRetriever` 来融合这两种检索器。在 `hybrid_retriever.py` 中，我们先创建向量检索器和 BM25 检索器，然后根据查询长度计算权重，最后创建 `EnsembleRetriever` 返回。

这样做的好处是能够兼顾精确匹配和语义理解，提升整体的检索召回率和准确率。"

**代码参考：**

```python
# 文件路径: backend/app/rag/retrievers/hybrid_retriever.py
class HybridRetriever:
    async def get_retriever(self, query=None, user_id=None):
        filter_dict = {'user_id': user_id}
        vector_retriever = self.vectors_store.as_retriever(
            search_type='similarity',
            search_kwargs={'k': chroma_config['k'], 'filter': filter_dict}
        )
        
        bm25_retriever = await self.get_bm25_retriever(user_id)
        
        if bm25_retriever:
            weights = await self.get_dynamic_weights(query)
            return EnsembleRetriever(
                retrievers=[vector_retriever, bm25_retriever],
                weights=weights
            )
        return vector_retriever
    
    @staticmethod
    async def get_dynamic_weights(query=None):
        if not query:
            return [0.5, 0.5]
        
        query_length = len(query)
        if query_length > 50:
            return [0.7, 0.3]  # 长查询更依赖向量检索
        elif query_length < 20:
            return [0.3, 0.7]  # 短查询更依赖BM25
        return [0.5, 0.5]
```

**面试话术：**

"面试官您好，我们项目中的文本分块策略主要基于 `RecursiveCharacterTextSplitter`。具体来说，我们设置块大小为 500 字符，块重叠为 50 字符。这样设置的原因是，500 字符的块大小既能保证每个块有足够的上下文信息，又不会太长导致向量表示不够精确。50 字符的重叠可以确保相邻块之间有一定的上下文连续性，避免信息丢失。

这样的配置和流程能够有效地将文档向量化存储，支持后续的高效检索。"

### 面试题 6：RAG 的文档重排序机制是如何实现的？

**面试话术：**

"在我们的 RAG 流程中，文档重排序是一个很重要的环节。我们使用了交叉编码器（Cross-Encoder）模型来进行重排序。

具体来说，当检索器返回一组文档后，我们会将每个文档与用户查询配对，然后使用交叉编码器模型对每一对进行评分。评分越高，说明文档与查询的相关性越强。

实现上，我们在 `reorder_service.py` 中创建了 `reorder_documents` 方法，该方法接收查询和文档列表，使用交叉编码器预测每个文档的分数，然后按分数降序排序。

在 RAG 服务中，我们先检索文档，然后对检索结果进行重排序，最后将排序后的文档作为上下文传递给 LLM 生成最终答案。

使用交叉编码器进行重排序的好处是，它能够更好地理解查询和文档之间的语义相关性，相比单纯的向量相似度，能够提供更准确的排序结果，从而提升整体的回答质量。"

**代码参考：**

```python
# 文件路径: backend/app/rag/reorder_service.py
class ReorderService:
    async def reorder_documents(self, query, documents):
        scores = await self.cross_encoder.predict(
            [(query, doc) for doc in documents]
        )
        
        sorted_docs = sorted(
            zip(documents, scores),
            key=lambda x: x[1],
            reverse=True
        )
        
        return {
            "success": True,
            "documents": [{"document": doc, "score": score} for doc, score in sorted_docs]
        }

# 文件路径: backend/app/rag/rag_service.py
async def get_documents_and_summary(self, query):
    documents = await self.retrieve_document(query)
    document_contents = [doc.page_content for doc in documents]
    reordered_documents = await self.reorder_documents(query, document_contents)
    
    final_summary = await self.chain.ainvoke({
        "input": query,
        "context": "\n".join(reordered_documents)
    })
```

## 第三章：HyDE 技术实现

### 面试题 7：什么是 HyDE 技术？项目中是如何实现的？

**面试话术：**

"面试官您好，HyDE（Hypothetical Document Embeddings）是一种提升检索召回率的技术。它的核心思想是，先使用 LLM 根据用户查询生成一个'假设性答案'，然后将这个假设性答案作为查询进行向量检索，这样可以找到与假设性答案最相关的文档。

在我们项目中，HyDE 的实现主要分为两个步骤：

第一步是生成假设性文档。我们在 `rag_service.py` 中定义了 `generate_hypothetical_document` 方法，使用一个精心设计的 prompt 来引导 LLM 生成假设性回答。Prompt 的内容大致是：'基于以下问题，生成一个详细的假设性回答，我将根据你的这个假设性回答在向量数据库里检索文档'。

第二步是使用假设性文档进行检索。在 `retrieve_document` 方法中，我们先调用 `generate_hypothetical_document` 生成假设性文档，然后将其作为查询传递给检索器进行向量检索。

HyDE 的优势在于，它可以帮助解决语义鸿沟问题。有时候用户的提问方式和知识库中文档的表达方式可能不同，直接检索可能无法找到相关文档。通过生成假设性文档，可以桥接这种语义差异，提升检索召回率。"

**代码参考：**

```python
# 文件路径: backend/app/rag/rag_service.py
class RagService:
    def __init__(self, user_id=None):
        self.hyde_prompt_template = PromptTemplate.from_template(
            "基于以下问题，生成一个详细的假设性回答，"
            "我将根据你的这个假设性回答在向量数据库里检索文档：\n\n"
            "问题：{query}\n\n"
            "假设性回答："
        )
    
    async def generate_hypothetical_document(self, query: str) -> str:
        hyde_chain = (
            self.hyde_prompt_template
            | self.chat_model
            | StrOutputParser()
        )
        hypothetical_doc = await hyde_chain.ainvoke({"query": query})
        return hypothetical_doc
    
    async def retrieve_document(self, query: str) -> list:
        hypothetical_doc = await self.generate_hypothetical_document(query)
        documents = await self.retriever.ainvoke(hypothetical_doc)
        return documents
```

### 面试题 8：HyDE 技术解决了什么问题？在冷启动场景下有什么优势？

**面试话术：**

"HyDE 技术主要解决了三个核心问题：

首先是语义鸿沟问题。用户提问的表达方式和知识库中文档的表达方式可能存在差异，直接检索可能无法找到相关文档。HyDE 通过生成假设性文档，可以将用户查询转换为更接近文档风格的表达，从而找到更多相关文档。

其次是关键词不匹配问题。用户可能使用不同的词汇来描述相同的概念，这会导致关键词检索失败。HyDE 生成的假设性回答会包含更多与目标文档匹配的关键词，提升检索成功率。

第三个是冷启动问题。当知识库中文档数量较少时，直接检索的召回率可能很低。HyDE 通过生成假设性文档，可以扩大检索范围，在有限的文档中找到更相关的内容。

在冷启动场景下，HyDE 的优势非常明显。比如当知识库只有少量文档时，传统检索可能无法找到足够的相关文档，但 HyDE 可以通过生成假设性回答，更好地利用现有的文档资源，提升回答质量。

另外，我们在实现中还考虑了回退机制。如果 HyDE 生成失败，我们会回退到使用原始查询进行检索，确保系统的稳定性。"

**代码参考：**

```python
async def generate_hypothetical_document(self, query: str) -> str:
    try:
        hypothetical_doc = await hyde_chain.ainvoke({"query": query})
        return hypothetical_doc
    except Exception as e:
        logger.error(f"【HyDE】生成失败: {e}")
        return query  # 失败时回退到原始查询
```

## 第四章：Redis 缓存体系

### 面试题 9：项目中 Redis 缓存体系的设计思路是什么？实现了哪些缓存策略？

**面试话术：**

"面试官您好，我们项目中的 Redis 缓存体系主要基于装饰器模式来实现。核心思路是通过 `@cache_with_redis` 装饰器自动缓存函数的返回值，从而减少数据库查询次数，提升系统响应速度。

具体来说，我们实现了以下几种缓存策略：

首先是会话列表缓存，过期时间设置为 3600 秒。当用户获取会话列表时，会先检查缓存，如果缓存命中就直接返回，否则查询数据库并更新缓存。

其次是文档元数据缓存，这个缓存永不过期，因为文档元数据一旦创建就不会频繁变化，只有在文件删除时才会手动删除缓存。

第三是限流计数缓存，用于 API 访问控制，按照时间窗口进行过期。

第四是检索结果缓存，过期时间设置为 300 秒，用于缓存 RAG 检索结果，减少重复检索。

实现上，我们在 `redis_decorator.py` 中创建了 `RedisCache` 类，提供 `get_or_set` 方法来实现缓存逻辑。装饰器会自动生成缓存键，并在函数执行前后进行缓存操作。

这样的设计使得缓存逻辑与业务逻辑解耦，使用起来非常方便，只需要在需要缓存的函数上添加装饰器即可。"

**代码参考：**

```python
# 文件路径: backend/app/cache/redis_decorator.py
class RedisCache(Generic[T]):
    @staticmethod
    async def get_or_set(key, func, *args, expire=3600, **kwargs):
        cached_data = await get_redis_cache_json(key)
        if cached_data is None:
            cached_data = await get_redis_cache_str(key)
        
        if cached_data is not None:
            return cached_data
        
        result = await func(*args, **kwargs)
        serializable_result = convert_to_serializable(result)
        await set_redis_cache(key, serializable_result, expire)
        
        return result

def cache_with_redis(prefix: str, expire: int = 3600):
    def decorator(func: Callable[..., T]) -> Callable[..., T]:
        @wraps(func)
        async def wrapper(*args, **kwargs):
            key = RedisCache.cache_key(prefix, *args, **kwargs)
            return await RedisCache.get_or_set(key, func, *args, expire=expire, **kwargs)
        return wrapper
    return decorator
```

### 面试题 10：缓存键的生成策略是怎样的？如何避免缓存键冲突？

**面试话术：**

"在我们的缓存体系中，缓存键的生成策略是确保缓存正确性和避免冲突的关键。

我们的缓存键格式是：`prefix:arg1:arg2:key1:value1:key2:value2`。

具体来说，我们会按照以下步骤生成缓存键：

首先，使用一个统一的前缀来区分不同类型的缓存，比如 `user_sessions`、`document_metadata` 等。

然后，添加位置参数，但会排除数据库会话等特殊对象，因为这些对象无法序列化且可能每次都不同。

接下来，添加关键字参数，并按字母排序，这样可以保证相同参数的不同调用顺序生成相同的缓存键。

最后，将所有部分用冒号连接起来，形成最终的缓存键。

为了避免缓存键冲突，我们采取了以下措施：

1. 使用统一的前缀命名规范，不同类型的缓存使用不同的前缀；
2. 参数按字母排序，确保缓存键的一致性；
3. 排除数据库会话等可变对象，避免生成不稳定的缓存键；
4. 使用用户 ID 作为缓存键的一部分，确保不同用户的缓存相互隔离。

这样的策略可以有效地避免缓存键冲突，确保缓存的正确性。"

**代码参考：**

```python
# 文件路径: backend/app/cache/redis_decorator.py
@staticmethod
def cache_key(prefix: str, *args, **kwargs) -> str:
    parts = [prefix]
    
    for arg in args:
        if arg is not None and not hasattr(arg, 'execute'):
            parts.append(str(arg))
    
    for key, value in sorted(kwargs.items()):
        if value is not None and key != 'db':
            parts.append(f"{key}:{value}")
    
    return ":".join(parts)
```

### 面试题 11：缓存失效机制是如何设计的？如何保证缓存与数据库的数据一致性？

**面试话术：**

"我们的缓存失效机制主要有两种方式：时间过期失效和主动删除失效。

时间过期失效是指为每个缓存设置一个过期时间，当缓存过期后会自动被 Redis 删除。比如会话列表缓存设置为 3600 秒过期，检索结果缓存设置为 300 秒过期。

主动删除失效是指在数据更新时，主动删除相关的缓存，这样下次读取时会从数据库获取最新数据并重建缓存。

在保证缓存与数据库一致性方面，我们采用的策略是'先写数据库，后删缓存'。具体流程如下：

当需要更新数据时，首先更新 MySQL 数据库，然后删除对应的 Redis 缓存。下次读取时，由于缓存已被删除，系统会从数据库读取最新数据并更新缓存。

这种策略的优点是简单可靠，虽然可能存在短暂的不一致（更新数据库后、删除缓存前的这段时间），但在大多数场景下是可以接受的。

另外，对于需要强一致性的场景，我们可以采用'先删缓存，后写数据库'的策略，但这可能会导致多次数据库查询，需要根据具体场景进行权衡。"

### 面试题 15：项目中 SSE 接口是如何定义的？如何实现实时流式推送？

**面试话术：**

"在我们项目中，SSE 接口的定义非常简洁。使用 FastAPI 的 `StreamingResponse` 来返回流式响应。

具体来说，在 `chat.py` 中，我们定义了 `query_stream` 端点，它接收查询请求和用户 ID，然后返回一个 `StreamingResponse` 对象。`StreamingResponse` 的内容是一个异步生成器 `get_agent_stream_response`。

在 `get_agent_stream_response` 函数中，我们使用 `agent_executor.astream` 方法来异步流式执行 Agent。当获取到输出时，我们会逐字发送，实现打字机效果。同时，我们会发送不同类型的事件，包括 `response`（响应内容）、`done`（响应结束）和 `error`（错误信息）。

这样的实现使得客户端可以实时接收 AI 的响应，而不需要等待整个响应完成，大大提升了用户体验。"

**代码参考：**

```python
# 文件路径: backend/app/router/chat.py
@chat_router.post("/agent/query/stream")
async def query_stream(
    request: QueryRequest,
    user_id: str = Depends(get_current_user_id),
    _: None = Depends(rate_limit(limit=10, window=60))
):
    session_id = request.session_id or str(uuid.uuid4())
    
    return StreamingResponse(
        get_agent_stream_response(request.query, session_id, user_id),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "Connection": "keep-alive"}
    )

# 文件路径: backend/app/agent/agent.py
async def get_agent_stream_response(query, session_id, user_id) -> AsyncGenerator:
    yield f"data: {json.dumps({'type': 'response', 'content': '', 'session_id': session_id})}\n\n"
    
    async for chunk in agent_executor.astream({
        "input": query,
        "chat_history": chat_history,
        "system_prompt": system_prompt
    }):
        if "output" in chunk:
            for char in chunk["output"]:
                yield f"data: {json.dumps({'type': 'response', 'content': char})}\n\n"
                await asyncio.sleep(0.02)
    
    yield f"data: {json.dumps({'type': 'done', 'session_id': session_id})}\n\n"
```

### 面试题 16：SSE 支持哪些事件类型？前端如何接收和处理流式数据？

**面试话术：**

"我们的 SSE 实现支持三种事件类型：

第一种是 `response` 类型，用于传输 AI 的响应内容。每次发送一个字符，实现打字机效果。

第二种是 `done` 类型，表示响应结束，包含会话 ID。

第三种是 `error` 类型，用于传输错误信息。

前端处理流程如下：使用 `EventSource` 连接到 SSE 端点，设置请求头和请求体。然后监听 `onmessage` 事件，解析接收到的数据，根据事件类型进行不同的处理。

对于 `response` 类型，将内容追加到响应字符串中，并更新 UI。

对于 `done` 类型，关闭连接并保存会话。

对于 `error` 类型，关闭连接并显示错误信息。

这样的设计使得前端可以实时更新 UI，提供流畅的用户体验。"

**代码参考（前端 Vue3）：**

```javascript
async function startStreamingChat(query, sessionId) {
  const eventSource = new EventSource('/chat/agent/query/stream', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`
    },
    body: JSON.stringify({ query, session_id: sessionId })
  });
  
  let fullResponse = '';
  
  eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    
    switch (data.type) {
      case 'response':
        fullResponse += data.content;
        updateChatUI(fullResponse);
        break;
      case 'done':
        eventSource.close();
        saveSession(sessionId);
        break;
      case 'error':
        console.error('Error:', data.content);
        eventSource.close();
        showError(data.content);
        break;
    }
  };
}
```
